# ADR-018: AIWG.md Hook File for Context Decoupling

**Status**: Accepted
**Date**: 2026-03-23
**Deciders**: Architecture Designer, Project Owner
**Context**: AIWG Context Injection Architecture

---

## Context and Problem Statement

AIWG injects framework context (agent instructions, workflow guidance, vocabulary, constraints) into AI coding assistant sessions via CLAUDE.md. As AIWG usage grows, CLAUDE.md becomes bloated: it simultaneously serves as a project-instructions file (owned by the developer) and an AIWG-framework-context file (owned by AIWG). These concerns have different authors, update cadences, and lifecycle owners.

Symptoms of the problem:
- CLAUDE.md files exceeding 500 lines in active AIWG projects
- Developers reluctant to edit CLAUDE.md because they may break AIWG injection
- AIWG updates require editing developer-owned CLAUDE.md
- No mechanism to disable AIWG context without editing CLAUDE.md manually

Options considered:
1. **Continue using CLAUDE.md** — current approach; single file, growing bloat
2. **Dedicated AIWG.md hook file** — AIWG context lives in a separate file, referenced from CLAUDE.md via import or @-mention
3. **Per-framework context files** — one file per installed framework (SDLC.md, MARKETING.md, etc.)
4. **Environment variable injection** — pass AIWG context via environment, not files

## Decision Drivers

- **Separation of concerns**: Developer instructions and framework context should be independently editable
- **Lifecycle ownership**: AIWG should be able to update its own context file without touching developer files
- **Enable/disable**: Framework context should be disableable without manual file editing
- **Regeneration**: `aiwg hook-regenerate` should be safe to run without clobbering developer customizations
- **Convention compatibility**: Solution must work across all 8 AIWG-supported platforms

## Decision

Decouple AIWG framework context from CLAUDE.md into a dedicated **AIWG.md hook file**, with a defined lifecycle: `hook-enable` → `hook-disable` → `hook-regenerate`.

CLAUDE.md retains a single @-reference to AIWG.md:
```
@AIWG.md
```

AIWG.md is owned and managed by AIWG tooling. CLAUDE.md is owned by the developer.

## Rationale

1. **Clean separation**: CLAUDE.md remains the developer's file; AIWG.md is AIWG's file
2. **Safe regeneration**: `aiwg hook-regenerate` rewrites AIWG.md without touching CLAUDE.md
3. **One-line integration**: `@AIWG.md` in CLAUDE.md is minimal, obvious, and easy to remove
4. **Lifecycle commands map to predictable outcomes**: enable/disable toggle the @-reference; regenerate refreshes content
5. **Platform generalization**: The pattern extends to other platforms (AGENTS.md, WARP.md) using platform-appropriate inclusion syntax

## Consequences

### Positive

- CLAUDE.md stays concise (developer's own instructions only)
- AIWG context can be updated or regenerated without developer involvement
- Disabling AIWG is a single command (`aiwg hook-disable`) rather than manual editing
- Per-framework content can be conditionally included in AIWG.md based on installed frameworks

### Negative

- Projects upgrading from pre-ADR-018 state must migrate: AIWG content extracted from CLAUDE.md into new AIWG.md
- Introduces a new file that developers must be aware of (though they should not edit it directly)
- Some platforms may not support @-style file inclusion natively (handled by platform-specific adapters)

### Risks

**Risk: Developers edit AIWG.md directly, then hook-regenerate overwrites their changes** (MEDIUM)
- **Mitigation**: `hook-regenerate` checks for local modifications and warns before overwriting; `--force` flag required to proceed
- **Acceptance**: Clear ownership boundary (AIWG.md = AIWG's file) documented in file header

## AIWG.md Structure

```markdown
<!-- AUTO-GENERATED BY AIWG — DO NOT EDIT DIRECTLY -->
<!-- To customize: edit CLAUDE.md. To update: aiwg hook-regenerate -->
<!-- To disable: aiwg hook-disable -->

# AIWG Framework Context
[Framework-injected content for installed frameworks]
```

## Lifecycle Commands

| Command | Action |
|---------|--------|
| `aiwg hook-enable` | Creates AIWG.md and adds `@AIWG.md` to CLAUDE.md |
| `aiwg hook-disable` | Removes `@AIWG.md` from CLAUDE.md (AIWG.md retained) |
| `aiwg hook-regenerate` | Rewrites AIWG.md content based on installed frameworks |

## Alternatives Considered

### Alternative 1: Continue using CLAUDE.md
Rejected: Unsustainable as framework grows. Developer/framework content ownership conflict has no clean resolution in a single file.

### Alternative 2: Per-framework context files (SDLC.md, MARKETING.md)
Rejected: Proliferates files; each new framework requires CLAUDE.md to reference another file. AIWG.md as a single aggregation point is simpler.

### Alternative 3: Environment variable injection
Rejected: Not portable across all 8 platforms; no file to audit or version control; breaks offline use.

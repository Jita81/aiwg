## `aiwg session` — Self-Healing Session Launcher

Starting an agentic session used to mean remembering a checklist: is aiwg up to date? Are the framework files deployed? Did I inject my MCP servers? `aiwg session` automates all of that.

### Pre-flight sequence

Every `aiwg session` run executes these steps before touching a provider:

1. **Version check** — compares installed vs latest; runs `npm install -g aiwg@latest` if stale
2. **Health check** — runs `aiwg doctor`; auto-repairs fixable issues via `aiwg sync`
3. **Deployment check** — reads `.aiwg/aiwg.config`; redeploys any framework files missing from the target provider
4. **MCP inject** (when `mcp` subcommand used) — runs `aiwg mcp inject --provider <p>`
5. **Launch** — spawns binary (claude, codex, opencode) or prints start instructions (cursor, windsurf, copilot, factory, warp)

IDE-integrated providers (cursor, windsurf, etc.) receive the same pre-flight. The only difference is the final step: they get context-aware start instructions instead of a terminal handoff.

### Auto-repair escalation

When `aiwg doctor` fails:

- **Strategy 1**: `aiwg sync` (update + redeploy)
- **Strategy 2**: Full `npm install -g aiwg@latest` + redeploy all frameworks
- **Unresolvable**: Surfaces `aiwg feedback --type bug` as the escape hatch

### Usage

```bash
# Default: full pre-flight then launch claude
aiwg session

# With MCP servers injected first
aiwg session mcp

# Explicit provider
aiwg session --provider codex
aiwg session --provider cursor       # IDE — prints instructions
aiwg session mcp --provider cursor   # Inject MCPs then print instructions

# Skip auto-repair (check only)
aiwg session --no-repair
```

### Provider resolution

Provider is resolved in this order:
1. `--provider` flag
2. `.aiwg/aiwg.config` → `providers[0]`
3. Default: `claude`

### Skills

Natural language also works. The new `session` skill in `aiwg-utils` maps phrases to commands:

| Phrase | Command |
|--------|---------|
| "start a session" | `aiwg session` |
| "launch claude" | `aiwg session --provider claude` |
| "start with my MCP servers" | `aiwg session mcp` |
| "launch codex and inject MCPs" | `aiwg session mcp --provider codex` |
| "just launch, skip the checks" | `aiwg session --no-repair` |

## `aiwg feedback` — GitHub Issue Submitter

When something breaks, filing an issue shouldn't require copying version strings and environment details by hand.

```bash
# Interactive — prompts for type, title, description
aiwg feedback

# Non-interactive
aiwg feedback --type bug \
  --title "doctor crashes in empty project" \
  --body "Running aiwg doctor in a new directory with no .aiwg causes an unhandled exception."

# Feature request
aiwg feedback --type feature --title "add --watch flag to aiwg index build"

# Doc gap
aiwg feedback --type doc --title "mcp inject workflow not documented"

# Skip context collection (privacy)
aiwg feedback --type bug --title "crash" --body "details" --no-context
```

### System context — automatic

Every submission includes a prefilled context block:

| Field | Source |
|-------|--------|
| aiwg version | `aiwg version` output |
| Node.js | `process.version` |
| OS | `os.type() + os.release()` |
| Arch | `os.arch()` |
| Provider | `.aiwg/aiwg.config` `providers[0]` |
| Frameworks | `.aiwg/aiwg.config` `installed` keys |
| Shell | `$SHELL` / `$COMSPEC` |

Omit with `--no-context`.

### Submission flow

1. `gh issue create --repo jmagly/aiwg` — if `gh` CLI is present
2. Browser URL with pre-filled title + body — if TTY
3. Print formatted body to stdout — fallback for non-interactive environments

### Alias: `aiwg report`

`aiwg report` is an alias for `aiwg feedback`.

### Doctor integration

`aiwg doctor` now surfaces `aiwg feedback --type bug` automatically when it finds issues it cannot auto-repair. `aiwg session` escalates to it at the end of its repair sequence for the same reason.

### Skills

The `feedback` skill in `aiwg-utils` extracts type, title, and body from conversation context:

| Phrase | Command |
|--------|---------|
| "report a bug" | `aiwg feedback --type bug` |
| "aiwg doctor crashes when .aiwg is missing" | Extracts type/title/body → non-interactive `aiwg feedback` |
| "request a feature for aiwg session" | `aiwg feedback --type feature --title "..."` |
| "the mcp inject docs are wrong" | `aiwg feedback --type doc` |

## `aiwg serve` WebSocket Fix (#851)

Sandbox WebSocket connections to `/ws/sandbox/:id` were returning 404 silently. The root cause: `@hono/node-server` v1.19.14 never shipped a `ws.js` export. The `createNodeWebSocket` call was wrapped in a `try/catch` that swallowed the `ERR_MODULE_NOT_FOUND`, leaving `upgradeWebSocket = null`. Every subsequent upgrade request fell through to a 404.

**Fix:** `serve.ts` now uses a `setupWebSockets()` function that attaches after `serve()` returns, imports `ws` dynamically, creates a `WebSocketServer({ noServer: true })`, and routes upgrade requests via URL regex:

- `/ws/sandbox/:sandboxId` → sandbox multiplexer
- `/ws/pty/:sessionId` → PTY stream

`ws` is added to `optionalDependencies` and to the auto-install list run on first `aiwg serve` launch. No manual installation required.

## ADR Template: 5 New Sections

The AIWG ADR template gains five new sections that address the most common gaps in architectural decision records — the things that cause an ADR to be unclear, unenforced, or unverifiable.

### Source Verification & Claim Tracking (Phase 1/2 boundary)

Always visible. A table of Claim | Source | Verified | Date forces authors to back up every factual assertion. Unverified claims block L2 acceptance.

```markdown
| Claim | Source | Verified | Date |
|-------|--------|----------|------|
| ws v8 handles 10k concurrent connections | ws README benchmarks | ✅ | 2026-04-10 |
| createNodeWebSocket ships in @hono/node-server | npm registry | ✅ | 2026-04-10 |
```

### Implementation Sketch (Phase 3)

Annotated code showing the key interaction, plus integration points and sharp edges — enough for a reviewer to spot architectural problems before implementation starts.

### Concurrency and Shared State Model (Phase 3)

Declares the concurrency model (single-threaded, actor, CSP, etc.), inventories every piece of shared mutable state, maps race conditions to mitigations, and explicitly scopes what is out of bounds.

### Testing Strategy (Phase 3)

Five layers: unit, integration, contract, performance, and a regression guard that prevents future changes from violating the decision silently.

### Definition of Done (Phase 3)

A five-level table: L1 Proposed → L2 Reviewed → L3 Accepted → L4 Implemented → L5 Verified. Each level has specific criteria and a blocking-items checklist. ADRs are no longer "done" when they're written — they're done when they're verified in production.

## Upgrading

```bash
npm install -g aiwg@latest

# Verify
aiwg version
aiwg doctor
```

No breaking changes. The new commands are additive. The ADR template additions are backwards-compatible — existing ADRs don't need the new sections unless they're being revised.

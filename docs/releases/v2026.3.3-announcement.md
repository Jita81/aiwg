# AIWG v2026.3.3 — "Identity & Infrastructure" Release

**Released**: 2026-03-23

This release introduces first-class agent identity files (SOUL.md), decouples AIWG context from provider config files via a dedicated hook file architecture, adds agent intelligence through constraint learning and domain grounding, builds out a comprehensive quality and metrics infrastructure, and ships a model evaluation suite for testing local and cloud models against AIWG's operational requirements.

## SOUL.md — Agent Identity Files

SOUL.md files give agents persistent identity: worldview, opinions, character traits, calibration examples. Unlike voice profiles (tone and style), SOUL.md encodes what an agent believes and how it reasons.

### Commands

```bash
# Create a SOUL.md for an agent
/soul-enable architecture-designer

# Check soul status across deployed agents
/soul-status

# Validate a soul file for completeness
/soul-validate .claude/agents/architecture-designer.soul.md

# Disable soul enforcement without deleting the file
/soul-disable architecture-designer
```

### Skills

```bash
# Enhance a vague or incomplete SOUL.md
soul-enhance architecture-designer.soul.md

# Apply soul identity to content generation (deeper than voice-apply)
soul-apply --agent security-auditor --content draft.md

# Merge multiple soul files with configurable conflict resolution
soul-blend test-engineer.soul.md security-auditor.soul.md --strategy weighted

# Bridge between soul and voice profiles
/voice-to-soul voices/technical-authority.md   # Generate soul from voice
/soul-to-voice architecture-designer.soul.md   # Extract voice from soul
```

### Deployment

Soul files deploy automatically alongside their companion agents when you run `aiwg use sdlc`. Each agent directory receives a `.soul.md` file on all 8 platforms.

Four SDLC agents ship with pre-built soul files:

- **test-engineer** — skeptical tester, integration-first, distrusts mocks
- **security-auditor** — assume breach, defense in depth, never trust inputs
- **architecture-designer** — systems thinker, monolith-first, explicit trade-offs
- **requirements-analyst** — assumption surfacer, always asks why, documents what's deferred

## AIWG.md Hook File Architecture

AIWG context is now decoupled from your provider config file (CLAUDE.md, WARP.md, etc.) into a dedicated `AIWG.md` loaded via an `@`-link directive. Your CLAUDE.md shrinks from 488+ lines to a 12-line wrapper.

### Commands

```bash
# Add @AIWG.md directive to your provider context file
/hook-enable

# Remove the directive without deleting AIWG.md
/hook-disable

# Show hook state across all 8 providers
/hook-status

# Rebuild AIWG.md from installed framework fragments
/hook-regenerate

# Migrate an existing full-inject installation to hook style
/migrate-hook
```

### How it works

`AIWG.md` is assembled from `contextContributions.hookFragment` entries in each installed framework's manifest. Install SDLC, run `hook-regenerate`, and AIWG.md updates automatically. Toggle AIWG context on/off without editing CLAUDE.md.

### Provider support

| Provider | Mechanism |
|----------|-----------|
| Claude Code | `@AIWG.md` directive in CLAUDE.md |
| Warp | `@AIWG-warp.md` directive in WARP.md |
| Cursor | `@AIWG-cursor.md` directive in rules file |
| OpenCode | `@AIWG-opencode.md` directive in config |
| Copilot | Full-inject (AGENTS.md, no native @-link) |
| Windsurf | Full-inject (no native @-link) |
| Factory | Full-inject (no native @-link) |
| Codex | Full-inject (no native @-link) |

## Agent Intelligence

### Constraint Learning from Feedback (#146)

Agents now accumulate domain constraints from structured feedback. The `FeedbackCollector` records per-agent feedback, clusters recurring errors, and proposes versioned constraints.

```
.aiwg/feedback/{agent}/
├── feedback.jsonl     # raw feedback stream
└── constraints.json   # derived constraint proposals with version history
```

Constraints are proposed automatically when a pattern appears 3+ times. Human review before promotion to the agent's active constraint set.

### Domain Grounding Agents (#184)

Four grounding agents inject verified domain knowledge into agent reasoning chains, reducing hallucination on domain-specific claims.

| Agent | Domain | Knowledge location |
|-------|--------|--------------------|
| `security-grounding-agent` | Security controls, CVEs, threat models | `.aiwg/knowledge/security/` |
| `performance-grounding-agent` | Benchmarks, SLAs, profiling patterns | `.aiwg/knowledge/performance/` |
| `compliance-grounding-agent` | Regulatory requirements, audit criteria | `.aiwg/knowledge/compliance/` |
| `technology-grounding-agent` | Stack-specific patterns, API behaviors | `.aiwg/knowledge/technology/` |

Each grounding agent exposes `verify_claim` and `retrieve_knowledge` interfaces, and uses keyword search across file-based knowledge stores.

## Quality and Metrics Infrastructure

Four new TypeScript modules with full unit test coverage:

### Token per artifact tracking (`src/metrics/artifact-metrics.ts`)

Measures token cost per artifact line. Baseline benchmark: 124 tokens/line for SDLC documents. Tracks by artifact type and phase for budget planning.

### Token budget management (`src/metrics/context-budget.ts`)

Enforces a 70/30 context/generation split for agent prompts. Raises budget warnings before context window overflow. Configurable per agent role.

### Pattern-based quality scoring (`src/quality/scoring.ts`)

Scores artifact quality against known-good patterns stored as JSON per artifact type:

```
src/quality/patterns/
├── adr.json
├── sad.json
├── test-plan.json
└── use-case.json
```

Returns structured quality report with section-level scores and improvement suggestions.

### Feedback accuracy measurement (`src/quality/feedback-ab.ts`, `feedback-tracker.ts`)

A/B testing infrastructure for measuring whether feedback actually improves subsequent generations. Tracks suggestion acceptance rates and quality delta.

## Model Evaluation Suite

New `tools/eval/` framework for evaluating any local or cloud model for AIWG compatibility before deploying agents.

### Usage

```bash
# Evaluate a local Ollama model
npx tsx tools/eval/src/index.ts llama3.1:8b --backend ollama

# Evaluate with verbose output
npx tsx tools/eval/src/index.ts qwen3.5:9b --backend ollama --verbose

# Evaluate a cloud model
npx tsx tools/eval/src/index.ts claude-haiku-4-5-20251001
```

### Scoring tiers

| Score | Tier | Suitable for |
|-------|------|-------------|
| 90+ | opus-equivalent | Complex orchestration, architecture design |
| 70–89 | sonnet-equivalent | Most SDLC agents |
| 50–69 | haiku-equivalent | Simple tasks, routing, summaries |
| <50 | not-recommended | — |

### Evaluation dimensions

6 dimensions, 9 initial test cases:

- **tool-use** — basic edit, multi-step chain, search + selection
- **instruction-following** — conventional commits, no-attribution enforcement
- **coding** — TypeScript function generation
- **structured-output** — JSON generation, YAML frontmatter
- **reasoning** — task decomposition
- **context-handling** — (extensible)

Results emit Markdown and JSON reports.

## Hermes MCP Integration

Full sidecar architecture for running AIWG with the Hermes agent framework.

### Architecture

```
Hermes → MCP server (sidecar) → AIWG
```

Hermes talks to AIWG's MCP server. AIWG is not embedded in Hermes — it's a sidecar. This keeps the token footprint minimal.

### Token-optimized templates

```
agentic/code/frameworks/sdlc-complete/templates/hermes/
├── AGENTS.md.aiwg-template   # 869-char routing rules (<1,000 char target)
├── hermes-config-minimal.yaml # 5-tool MCP whitelist (~3,000 token schema)
└── hermes-config-prompts.yaml # advanced config with prompt exposure
```

### Two-model strategy

Hermes works best with specialized models per task type:

| Task | Recommended model |
|------|------------------|
| Conversation, soul application | hermes3 |
| Coding, tool calls | qwen3.5:9b |

Qwen 3.5 9B is the best open-weight model for tool call accuracy at this scale.

### delegate_task skill

The `aiwg-orchestrate` skill exposes a `delegate_task` pattern that reduces parent context by 95% vs direct MCP calls (~200 tokens vs 3,000–8,000):

```yaml
# Use in Hermes agent config
skills:
  - aiwg-orchestrate
```

### Platforms field on skills

79 skills now declare `platforms: [claude-code, hermes]`. 17 Claude-specific skills (Task tool dependencies, `.claude/` path references) declare `platforms: [claude-code]` to prevent accidental Hermes deployment.

## YAML Metalanguage Schemas

Phase 1 of the YAML metalanguage RFC: JSON Schema definitions for declarative agent orchestration.

```
agentic/code/frameworks/sdlc-complete/schemas/metalanguage/
├── flow.schema.json    # Phase transition flows
├── agent.schema.json   # Agent definitions
├── rule.schema.json    # Enforcement rules
└── skill.schema.json   # Skill definitions
```

### Example: flow definition in YAML

```yaml
# inception-to-elaboration.flow.yaml
flow: inception-to-elaboration
version: "1.0"
phases:
  from: inception
  to: elaboration
artifacts:
  - type: sad
    template: software-architecture-doc-template.md
    reviewers: [security-architect, test-architect, technical-writer]
gate:
  criteria:
    - all_reviewers_approved
    - no_open_blockers
```

Target: 40–60% token reduction across the definition corpus when the full metalanguage is adopted.

## Hybrid Artifact Addressing

Artifacts in `.aiwg/` can now be referenced by file path or semantic URN from the same query interface.

### Address syntax

```
@.aiwg/requirements/UC-001.md          # explicit path
@?"authentication flow"                # semantic search
@.aiwg/requirements/?"login"           # scoped semantic search
@#security                             # tag filter
@elaboration:sad                       # phase:type filter
```

### Query engine

```typescript
// src/artifacts/hybrid-query.ts
const results = await queryArtifacts('@?"authentication"');
const deps = await findDependencies('.aiwg/requirements/UC-001.md');
const dependents = await findDependents('.aiwg/architecture/sad.md');
```

In-memory index built from `.aiwg/` frontmatter for sub-100ms query performance.

## Verbalized Sampling Addon

New `agentic/code/addons/verbalized-sampling/` addon for generating diverse, non-repetitive AI output.

```bash
aiwg use verbalized-sampling
```

Includes:
- **content-diversifier agent** — generates multiple variations with explicit diversity targets
- **diversity-tuning skill** — tune sampling parameters for a given task
- **3 prompt strategies**: `vs-standard`, `vs-cot` (chain-of-thought), `vs-multi` (multi-sample)
- **diversity-awareness rule** — flags when output variance is too low

## Factory Fix: $ARGUMENTS Injection

Factory's slash command system requires `$ARGUMENTS` in the prompt body to pass user input. Without it, anything typed after the command name was silently dropped and commands ran as static prompts.

`aiwg use sdlc --provider factory` now performs a deploy-time transform:

- Prepends `$ARGUMENTS` to the deployed command body
- Adds `argument-hint` frontmatter (preserving source value if present)

Source files in `agentic/code/` are unchanged. This is a deploy-time transform consistent with how Copilot commands are converted to YAML and how Warp aggregates into WARP.md.

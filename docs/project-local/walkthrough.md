# Walkthrough: Setting Up Project-Local Customization

This walkthrough creates a `.aiwg/.project/` container from scratch. By the end you will have:

- A `manifest.json` defining your project's namespace and asset directories
- One project-local agent
- One project-local skill
- Everything deploying cleanly via `aiwg use`

The example uses a research corpus project, but the pattern applies to any project.

---

## Prerequisites

- AIWG installed (`npm install -g aiwg`)
- A project with `.aiwg/` already initialized (run `aiwg use` at least once)

---

## Step 1 — Create the container directory

```bash
mkdir -p .aiwg/.project/agents
mkdir -p .aiwg/.project/skills/my-skill
```

---

## Step 2 — Write the manifest

Create `.aiwg/.project/manifest.json`:

```json
{
  "$schema": "https://aiwg.io/schemas/project-manifest.v1.json",
  "name": "my-project",
  "description": "Project-specific agents and skills for my-project",
  "version": "1.0.0",
  "namespace": "myns",
  "assets": {
    "agents": ["agents/"],
    "skills": ["skills/"]
  }
}
```

Choose a `namespace` that reflects your project domain. It becomes the subdirectory name in the platform dirs — `myns` → `.claude/agents/myns/`.

---

## Step 3 — Write an agent

Create `.aiwg/.project/agents/domain-expert.md`:

```markdown
---
name: Domain Expert
description: Project-specific assistant with deep knowledge of this domain
model: sonnet
tools: Read, Write, Bash, Glob, Grep
---

# Domain Expert

You are a Domain Expert for [project name]. You know the project's conventions,
file layout, and workflows in detail.

## Key Conventions

- Files are organized as [describe your structure]
- Identifiers follow the pattern [your pattern, e.g., REF-XXX]
- Workflows run via [your scripts/commands]

## Your Responsibilities

1. Answer questions about the project's domain
2. Guide users through the standard workflows
3. Help with file and artifact management

## Important Paths

- `[path]/` — [what it contains]
- `[path]/` — [what it contains]
```

Customize the content to your project. The agent's system prompt should encode everything a new team member would need to know about the domain — not generic advice, but the specific conventions, paths, and workflows that are unique to this project.

---

## Step 4 — Write a skill

Create `.aiwg/.project/skills/my-skill/SKILL.md`:

```markdown
---
platforms: [claude-code]
description: Run the standard [action] workflow for this project
namespace: myns
commandHint:
  argumentHint: [target --dry-run]
  allowedTools: Read, Write, Bash, Glob, Grep
  model: sonnet
  category: project-management
---

# My Skill

You run the standard [action] workflow for this project.

## Task

When invoked with `/my-skill [target]`:

### Step 1 — Validate inputs

Check that [preconditions] are met before proceeding.

### Step 2 — Run the workflow

[Describe the steps the skill executes]

```bash
# Example commands the skill runs
[command 1]
[command 2]
```

### Step 3 — Confirm output

Check that the expected output exists and report success:

```
Done: [what was created/updated]
```

## Argument Reference

| Argument | Description | Default |
|----------|-------------|---------|
| `target` | [what target means] | required |
| `--dry-run` | Show what would happen, make no changes | false |
```

---

## Step 5 — Deploy

```bash
aiwg use
```

AIWG deploys all installed frameworks and addons, then picks up `.aiwg/.project/` and deploys your project assets into the namespaced subdirectories.

After deployment you should see:

```
.claude/agents/myns/domain-expert.md
.claude/skills/myns/my-skill/SKILL.md
```

Verify:

```bash
ls .claude/agents/myns/
ls .claude/skills/myns/
```

---

## Step 6 — Test the agent and skill

Open your AI coding assistant and confirm the agent and skill are available:

```
Use the Domain Expert agent to explain this project's file layout.
```

```
/my-skill --dry-run
```

---

## Step 7 — Commit the source, not the platform dirs

Add `.aiwg/.project/` to git:

```bash
git add .aiwg/.project/
git commit -m "feat: add project-local AIWG customization container"
```

Platform directories (`.claude/`, `.codex/`, etc.) can remain gitignored — they are fully regenerated by `aiwg use`.

If your team uses multiple platforms (Claude Code + Cursor + Copilot), each member's `aiwg use` call deploys to their platform automatically.

---

## Proving it's expendable

To verify the workflow is fully regenerable:

```bash
rm -rf .claude/
aiwg use
ls .claude/agents/myns/   # ← should be back
ls .claude/skills/myns/   # ← should be back
```

If both directories are restored, your project-local customization is correctly woven in.

---

## Best Practices

### Keep agents focused

Project-local agents should encode what is unique to your project — specific file layouts, naming conventions, domain concepts. Do not duplicate AIWG's built-in agents.

### Use the namespace consistently

Choose a namespace and keep it stable. Changing the namespace in `manifest.json` will change where assets deploy, which can confuse team members who have the old paths cached.

### One skill per workflow

Skills work best when they do one thing end-to-end. A skill named `paper-intake` that handles the full induction workflow is better than a generic `utilities` skill that does many things.

### Reference project structure from agents

Agents and skills deployed from `.aiwg/.project/` can freely reference paths within your project. Unlike AIWG's built-in agents (which must work in any project), your project-local agents can hard-code paths like `documentation/references/` because they only ever run in this project.

### Add scripts for non-trivial operations

If your skill needs to run a complex shell command, put it in `.aiwg/.project/scripts/` and reference it from the skill. This makes the skill cleaner and lets you test the script independently.

---

## Adding More Skills

Add new skill directories alongside the first:

```bash
mkdir .aiwg/.project/skills/another-skill
# create .aiwg/.project/skills/another-skill/SKILL.md
aiwg use  # redeploys everything, picks up new skill
```

---

## Troubleshooting

**Skill not appearing after `aiwg use`**

- Check that the skill directory has a `SKILL.md` file (the subdirectory alone is not enough)
- Verify the `skills/` directory is listed in `manifest.json` `assets.skills`
- Check `aiwg doctor` for deployment errors

**Agent not appearing**

- Verify the `.md` file is in a directory listed in `manifest.json` `assets.agents`
- Check the frontmatter is valid YAML (a syntax error silently prevents deployment)

**Namespace collision**

- If another tool is already using your chosen namespace, rename it in `manifest.json` and redeploy
